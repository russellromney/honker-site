If your app already owns the SQLite connection via an ORM, you don't have to hand it over to Honker. Load the `libhonker_ext` extension on each connection and call the [`honker_*` SQL functions](/reference/extension/) from inside your ORM's transaction. The enqueue commits atomically with your business write, which is the whole point.

The split looks like this:

- **Writes** go through your ORM. `SELECT honker_enqueue(...)` inside the session's transaction is one row in the `_honker_live` table, committed with your `INSERT INTO orders` — one COMMIT, all-or-nothing.
- **Workers** run in a separate process and use the honker binding directly (`honker.open("app.db")`, `db.queue("emails").claim(...)`). The WAL watcher wakes on commits from any connection to the file, including the ones your ORM just made.

You don't need the ORM to handle any worker/consumer code. Honker's binding gives you that for free. The ORM is just how the write side gets into the same file.

## What you need

1. **The compiled extension** (`libhonker_ext.dylib` / `.so` / `.dll`). Download from [releases](https://github.com/russellromney/honker/releases/latest) or `cargo build --release -p honker-extension`.
2. **Your SQLite driver must support runtime-loadable extensions.** This is a build flag; it's on for most distros and off for a few. Specifics per language below.
3. **WAL mode.** `honker_bootstrap()` refuses to run otherwise — see [the design notes](/#wal-only-by-design).

On every connection you open, once:

```sql
PRAGMA journal_mode = WAL;
SELECT honker_bootstrap();
```

Both are idempotent. Run them in the ORM's connect hook so you don't have to think about it.

## Python

<Tabs>
  <TabItem label="SQLAlchemy">
```python
from sqlalchemy import create_engine, event, text
from sqlalchemy.orm import Session

engine = create_engine("sqlite:///app.db")

@event.listens_for(engine, "connect")
def _load_honker(conn, _):
    conn.enable_load_extension(True)
    conn.load_extension("/path/to/libhonker_ext")
    conn.execute("PRAGMA journal_mode=WAL")
    conn.execute("SELECT honker_bootstrap()")

# Atomic: the enqueue commits with the order, or neither lands.
with Session(engine) as s, s.begin():
    s.add(Order(user_id=42))
    s.execute(
        text("SELECT honker_enqueue(:q, :p, NULL, NULL, 0, 3, NULL)"),
        {"q": "emails", "p": '{"to":"alice@example.com"}'},
    )
```

Workers run elsewhere:

```python
import asyncio, honker

db = honker.open("app.db")
async def run():
    async for job in db.queue("emails").claim("worker-1"):
        await send_email(job.payload); job.ack()
asyncio.run(run())
```

```python
# settings.py

    "default": {
        "ENGINE": "django.db.backends.sqlite3",

from django.db.backends.signals import connection_created
from django.dispatch import receiver

@receiver(connection_created)
def _load_honker(sender, connection, **kwargs):
```

Then in a view, call `honker_enqueue` inside `transaction.atomic`:

```python
from django.db import transaction, connection

with transaction.atomic():
    Order.objects.create(user_id=42)
    with connection.cursor() as cur:
        cur.execute(
            "SELECT honker_enqueue(%s, %s, NULL, NULL, 0, 3, NULL)",
            ["emails", '{"to":"alice@example.com"}'],
        )
```

Run the worker as a management command (same pattern as Celery) using `honker.open(settings.DATABASES["default"]["NAME"])`.

  <TabItem label="Prisma">
Prisma **currently can't load SQLite extensions.** Its SQLite build ships without `SQLITE_ENABLE_LOAD_EXTENSION` and its query engine doesn't expose the underlying connection. There's [an open feature request](https://github.com/prisma/prisma/issues/20271) tracking this.

Options if you want Honker + Prisma on the same app:

1. **Separate Honker connection.** Let Prisma own its connection; open a second connection to the same `.db` file via `better-sqlite3` just for the honker calls. You lose atomicity between the business write and the enqueue — the point of this guide — so this is more "Honker alongside Prisma" than "Honker with Prisma."
2. **Move the write side off Prisma.** If the enqueue-atomic-with-business-write property matters, wrap the affected writes in a `better-sqlite3` or Drizzle transaction and keep Prisma for everything else.

No great answer here yet. If Prisma ships extension support, this page will shorten.

</Tabs>

## Go (GORM, `database/sql`)

`mattn/go-sqlite3` has a declarative `Extensions` field on the driver. Register a driver variant once, then open connections through it:

```go
import (
    "database/sql"
    sqlite3 "github.com/mattn/go-sqlite3"
)

func init() {
    sql.Register("sqlite3_honker", &sqlite3.SQLiteDriver{
        Extensions: []string{"/path/to/libhonker_ext"},
    })
}

db, _ := sql.Open("sqlite3_honker", "file:app.db?_journal_mode=WAL")
db.Exec("SELECT honker_bootstrap()")

tx, _ := db.Begin()
tx.Exec("INSERT INTO orders (user_id) VALUES (?)", 42)
tx.Exec("SELECT honker_enqueue(?, ?, NULL, NULL, 0, 3, NULL)",
    "emails", `{"to":"alice@example.com"}`)
tx.Commit()
```

GORM wraps `database/sql` — feed it the same registered driver name:

```go
import (
    "gorm.io/driver/sqlite"
    "gorm.io/gorm"
)

db, _ := gorm.Open(
    sqlite.Dialector{DriverName: "sqlite3_honker", DSN: "file:app.db?_journal_mode=WAL"},
    &gorm.Config{},
)
```

Extension loading is **on by default** in `mattn/go-sqlite3`. The opt-out build tag is `sqlite_omit_load_extension`; don't set it.

## Hide the raw SQL with a tiny wrapper

Typing `session.execute(text("SELECT honker_enqueue(:q, :p, NULL, NULL, 0, 3, NULL)"), {...})` at 50 call sites reads like ass. Ten lines of your own code fixes it — and shipping the wrapper yourself beats any library helper because you can shape it to match your app's idioms.

### SQLAlchemy

Thin `Queue` class that takes a Session and looks like the native `honker.Queue`:

```python
# yourapp/honker_ext.py
import json
from typing import Any, Optional
from sqlalchemy import text
from sqlalchemy.orm import Session


class Queue:
    def __init__(self, name: str, *, max_attempts: int = 3):
        self.name = name
        self.max_attempts = max_attempts

    def enqueue(
        self,
        s: Session,
        payload: Any,
        *,
        delay: Optional[int] = None,
        priority: int = 0,
    ) -> int:
        return s.execute(
            text(
                "SELECT honker_enqueue("
                " :q, :p, NULL, :d, :pr, :ma, NULL"
                ") AS id"
            ),
            {
                "q": self.name,
                "p": json.dumps(payload),
                "d": delay,
                "pr": priority,
                "ma": self.max_attempts,
            },
        ).scalar_one()


def notify(s: Session, channel: str, payload: Any = None) -> None:
    s.execute(
        text("SELECT notify(:c, :p)"),
        {"c": channel, "p": json.dumps(payload) if payload is not None else None},
    )
```

Now the call sites read the way you'd write them with Honker's native API:

```python
from yourapp.honker_ext import Queue, notify

emails = Queue("emails")

with Session(engine) as s, s.begin():
    s.add(Order(user_id=42))
    emails.enqueue(s, {"to": "alice@example.com"})
    notify(s, "orders", {"id": 42})
```

### SQLModel + Pydantic (typed payloads)

SQLModel's `Session` inherits from SQLAlchemy's, so the wiring is identical — but you can push further and use a Pydantic model as the payload schema, giving you validation at enqueue and a typed `parse()` for the worker:

```python
# yourapp/honker_ext.py
from typing import Generic, Type, TypeVar, Optional
from pydantic import BaseModel
from sqlalchemy import text
from sqlmodel import Session

T = TypeVar("T", bound=BaseModel)


class TypedQueue(Generic[T]):
    def __init__(self, name: str, model: Type[T], *, max_attempts: int = 3):
        self.name = name
        self.model = model
        self.max_attempts = max_attempts

    def enqueue(
        self,
        s: Session,
        payload: T,
        *,
        delay: Optional[int] = None,
        priority: int = 0,
    ) -> int:
        return s.execute(
            text(
                "SELECT honker_enqueue("
                " :q, :p, NULL, :d, :pr, :ma, NULL"
                ") AS id"
            ),
            {
                "q": self.name,
                "p": payload.model_dump_json(),
                "d": delay,
                "pr": priority,
                "ma": self.max_attempts,
            },
        ).scalar_one()

    def parse(self, raw: str) -> T:
        """Worker-side: validate a job payload back into the model."""
        return self.model.model_validate_json(raw)
```

Call sites get full IDE completion, Pydantic validation at enqueue, and a typed payload on the worker end:

```python
from pydantic import BaseModel, EmailStr
from yourapp.honker_ext import TypedQueue

class EmailJob(BaseModel):
    to: EmailStr
    subject: str
    body: str

emails = TypedQueue("emails", EmailJob)

# Caller — Pydantic rejects bad input before it ever hits SQLite.
with Session(engine) as s, s.begin():
    s.add(Order(user_id=42))
    emails.enqueue(s, EmailJob(
        to="alice@example.com", subject="Hi", body="..."))

# Worker (separate process, using the honker binding directly):
import honker
db = honker.open("app.db")
async for job in db.queue("emails").claim("worker-1"):
    email: EmailJob = emails.parse(job.payload)  # Pydantic validates
    await send(email.to, email.subject, email.body)
    job.ack()
```

The wrapper is yours — tune it. Add a `stream_publish(topic, payload)` helper if you use streams. Register a SQLAlchemy `event.listens_for(..., "after_flush")` hook if you want queued enqueues to batch with the flush. Shape it however the rest of your app is shaped.

### Drizzle / Kysely

```ts
// yourapp/honker-ext.ts
import { sql } from 'drizzle-orm';
import type { BetterSQLite3Database } from 'drizzle-orm/better-sqlite3';

export class Queue<T> {
  constructor(
    public readonly name: string,
    private readonly maxAttempts: number = 3,
  ) {}

  enqueue(
    db: BetterSQLite3Database,
    payload: T,
    opts: { delay?: number; priority?: number } = {},
  ): number {
    const row = db.get<{ id: number }>(sql`
      SELECT honker_enqueue(
        ${this.name},
        ${JSON.stringify(payload)},
        NULL,
        ${opts.delay ?? null},
        ${opts.priority ?? 0},
        ${this.maxAttempts},
        NULL
      ) AS id
    `);
    return row!.id;
  }
}

export function notify(
  db: BetterSQLite3Database,
  channel: string,
  payload: unknown = null,
): void {
  db.run(sql`
    SELECT notify(${channel}, ${payload === null ? null : JSON.stringify(payload)})
  `);
}
```

Call sites — typed payloads, atomic with your Drizzle writes:


### Go (`database/sql` + GORM)

An interface wide enough to accept `*sql.DB`, `*sql.Tx`, or `*sql.Conn` lets one helper serve transactional and non-transactional call sites:

```go
// yourapp/honker/honker.go
package honker

import (
    "database/sql"
    "encoding/json"
)

// DB is satisfied by *sql.DB, *sql.Tx, and *sql.Conn.
type DB interface {
    QueryRow(query string, args ...any) *sql.Row
    Exec(query string, args ...any) (sql.Result, error)
}

type EnqueueOpts struct {
    Delay    *int
    Priority int
}

type Queue struct {
    Name        string
    MaxAttempts int
}

func (q *Queue) Enqueue(db DB, payload any, opts EnqueueOpts) (int64, error) {
    p, err := json.Marshal(payload)
    if err != nil {
        return 0, err
    }
    maxAttempts := q.MaxAttempts
    if maxAttempts == 0 {
        maxAttempts = 3
    }
    var id int64
    err = db.QueryRow(
        "SELECT honker_enqueue(?, ?, NULL, ?, ?, ?, NULL)",
        q.Name, string(p), opts.Delay, opts.Priority, maxAttempts,
    ).Scan(&id)
    return id, err
}

func Notify(db DB, channel string, payload any) error {
    var body any
    if payload != nil {
        b, err := json.Marshal(payload)
        if err != nil {
            return err
        }
        body = string(b)
    }
    _, err := db.Exec("SELECT notify(?, ?)", channel, body)
    return err
}
```

Call sites — the wrapper is transaction-aware because `*sql.Tx` satisfies the `DB` interface:

```go
import "yourapp/honker"

var emails = &honker.Queue{Name: "emails"}

tx, _ := db.Begin()
tx.Exec("INSERT INTO orders (user_id) VALUES (?)", 42)
emails.Enqueue(tx, map[string]any{"to": "alice@example.com"}, honker.EnqueueOpts{})
honker.Notify(tx, "orders", map[string]any{"id": 42})
tx.Commit()
```

For GORM, reach through to the raw `*sql.Tx` inside `db.Transaction`:

```go
db.Transaction(func(tx *gorm.DB) error {
    if err := tx.Create(&order).Error; err != nil { return err }
    sqlTx := tx.Statement.ConnPool.(*sql.Tx)
    _, err := emails.Enqueue(sqlTx, EmailJob{To: "alice@example.com"}, honker.EnqueueOpts{})
    return err
})
```

### Plain better-sqlite3 / bun:sqlite (no ORM)

Not every TypeScript project has an ORM layer. A generic `Queue<T>` class hides the SQL just as well:

```ts
// yourapp/honker-ext.ts
import type Database from 'better-sqlite3';  // or: import type { Database } from 'bun:sqlite';

export class Queue<T> {
  constructor(
    private readonly db: Database.Database,
    public readonly name: string,
    private readonly maxAttempts: number = 3,
  ) {}

  enqueue(
    payload: T,
    opts: { delay?: number; priority?: number } = {},
  ): number {
    const row = this.db.prepare(
      "SELECT honker_enqueue(?, ?, NULL, ?, ?, ?, NULL) AS id"
    ).get(
      this.name,
      JSON.stringify(payload),
      opts.delay ?? null,
      opts.priority ?? 0,
      this.maxAttempts,
    ) as { id: number };
    return row.id;
  }
}

export function notify(
  db: Database.Database,
  channel: string,
  payload: unknown = null,
): void {
  db.prepare('SELECT notify(?, ?)').run(
    channel,
    payload === null ? null : JSON.stringify(payload),
  );
}
```

Call sites wrap business writes in a `db.transaction(...)`:

```ts
interface EmailJob { to: string; subject: string }

const emails = new Queue<EmailJob>(db, 'emails');

const createOrder = db.transaction((userId: number, email: EmailJob) => {
  db.prepare('INSERT INTO orders (user_id) VALUES (?)').run(userId);
  emails.enqueue(email);
});
createOrder(42, { to: 'alice@example.com', subject: 'Hi' });
```

### Rust (sqlx)

If you're writing a new Rust app, just use the [`honker` crate](https://crates.io/crates/honker) — it already gives you typed `Queue`, `Stream`, and `Database::transaction()`. Skip this whole section.

If you're retrofitting into a sqlx codebase and want to keep the sqlx types flowing:

```rust
// src/honker_ext.rs
use serde::Serialize;
use sqlx::{sqlite::SqliteRow, SqliteExecutor};

pub struct Queue {
    pub name: String,
    pub max_attempts: i32,
}

impl Queue {
    pub async fn enqueue<E, P>(
        &self,
        executor: E,
        payload: &P,
        delay: Option<i32>,
        priority: i32,
    ) -> sqlx::Result<i64>
    where
        E: for<'c> SqliteExecutor<'c>,
        P: Serialize,
    {
        let payload_json = serde_json::to_string(payload)
            .map_err(|e| sqlx::Error::Encode(Box::new(e)))?;
        let (id,): (i64,) = sqlx::query_as(
            "SELECT honker_enqueue($1, $2, NULL, $3, $4, $5, NULL)"
        )
        .bind(&self.name)
        .bind(payload_json)
        .bind(delay)
        .bind(priority)
        .bind(self.max_attempts)
        .fetch_one(executor)
        .await?;
        Ok(id)
    }
}

pub async fn notify<E, P>(executor: E, channel: &str, payload: Option<&P>) -> sqlx::Result<()>
where
    E: for<'c> SqliteExecutor<'c>,
    P: Serialize,
{
    let body = payload
        .map(serde_json::to_string)
        .transpose()
        .map_err(|e| sqlx::Error::Encode(Box::new(e)))?;
    sqlx::query("SELECT notify($1, $2)")
        .bind(channel)
        .bind(body)
        .execute(executor)
        .await?;
    Ok(())
}
```

`SqliteExecutor` is sqlx's abstraction over "thing you can run a query against" — it's satisfied by `&SqlitePool`, `&mut SqliteConnection`, and `&mut Transaction<'_, Sqlite>`, so one helper serves both transactional and non-transactional callers.

### Why this works

Ten-ish lines of wrapper code in your own project beats any shipped helper:

- No dependency surface — nothing to version-pin against SQLAlchemy 2.x / Drizzle / Ecto / Rails majors.
- You can shape the API to match your codebase (your naming, your types, your conventions).
- If Honker changes the `honker_enqueue` signature, you update one file, not your 50 call sites.
- No "helper package abandoned because the maintainer got bored."

## Caveats that apply everywhere

- **WAL is mandatory.** `honker_bootstrap()` refuses to install the schema on a non-WAL file. Set `journal_mode=WAL` before calling bootstrap.
- **One file, one machine.** SQLite's locking is for a single host. Two servers pointing at the same `.db` over NFS will corrupt it. Shard by file, or move to Postgres + Oban/pg-boss.
- **The workers are separate.** The ORM owns the write connection; the workers use `honker.open("app.db")` (or the equivalent in your language) in their own process. The WAL watcher fires across processes, so the worker wakes ~1 ms after your ORM commits.
- **Bootstrap is idempotent.** Calling `honker_bootstrap()` on every connect is safe but slightly wasteful. For production, run it once (initializer / migration) and drop it from the connect hook.

---
title: Bindings
description: How Honker thinks about cross-language parity, where Python is still ahead, and how maintained bindings should converge.
---

Honker has two different "parity" stories:

1. **Core runtime parity**
2. **Task-framework parity**

Those are not the same thing.

## Core runtime parity

This is the bar every maintained binding should hit.

It means the binding exposes the same shared SQLite-native runtime:

- queues
- jobs
- delayed `run_at`
- deadline wake
- streams
- notify/listen or a clearly documented update-watcher story
- scheduler
- results primitives
- locks / rate limits where the binding claims them

It also means the names should be as boringly similar as the language
allows:

- `run_at` / `runAt`
- `schedule`
- `cron` only as a compatibility alias
- `@every <n><unit>`

When users switch languages, the runtime concepts should feel the same.

## Task-framework parity

This is the richer layer Python already has:

- `@task`
- `@periodic_task`
- `TaskResult`
- `wait_result()`
- task registry
- worker CLI / task-runner ergonomics

That is a larger product surface than "thin wrapper over the shared
Honker runtime."

Today, Python is ahead here. That is real, and it is okay to say
plainly.

## Current strategy

Honker should push **core runtime parity** across all maintained
bindings.

Honker does **not** need to force full Python-style task-framework
parity into every language package.

That means:

- every maintained binding should converge on the same runtime behavior
- only a smaller set of bindings may carry the richer task layer

## Likely tiers

### Task-framework tier

These are the best candidates for the richer product layer:

- Python
- .NET
- maybe Node

### Core-runtime tier

These should still be great, but may stop at the shared runtime:

- Go
- Rust
- Ruby
- Elixir
- Bun
- C++

That is not second-class. It is just a narrower promise.

## Near-parity checklist

For each maintained binding, we want a boring checklist:

- queue enqueue / claim / ack / retry / fail / heartbeat
- delayed `run_at`
- deadline-aware wake
- streams with resume-after-crash semantics
- notify/listen or a clearly documented update-watcher equivalent
- scheduler add / remove / tick / soonest / run
- `schedule` as the canonical recurring name
- 5-field cron
- 6-field cron
- `@every <n><unit>`
- results primitives
- cross-language interop proof

If a binding has the richer task layer too, say so explicitly.
If it does not, say that explicitly too.

## Current maintained bindings

| Binding | Core runtime | Wake / scheduling | Results primitives | Task framework layer | Tier |
|---|---|---|---|---|---|
| Python | Complete | Complete | Complete | `@task`, `@periodic_task`, `TaskResult`, `wait_result()`, worker CLI | Task-framework |
| .NET | Complete | Complete | Complete | Not yet | Core-runtime today, likely task-framework candidate |
| Node | Complete | Complete | Complete | Not yet | Core-runtime today, maybe task-framework candidate |
| Rust | Complete | Complete | Complete | Not yet | Core-runtime |
| Go | Complete | Complete | Complete | Not yet | Core-runtime |
| Ruby | Complete | Complete | Complete | Not yet | Core-runtime |
| Elixir | Complete | Complete | Partial higher-level ergonomics | Not yet | Core-runtime |
| Bun | Complete | Complete | Complete | Not yet | Core-runtime |
| C++ | Complete | Complete | Basic runtime surface | Not yet | Core-runtime |

This table is intentionally boring. It is not trying to grade
languages on vibes. It is just tracking the promise each binding makes.

## What "complete" means here

For the **core runtime** columns, "complete" means the binding exposes
the shared Honker database/runtime surface and has proof around the
important time-trigger behavior:

- delayed `run_at`
- deadline wake
- scheduler / `schedule`
- 5-field cron
- 6-field cron
- `@every <n><unit>`

For the **task framework layer**, "not yet" means the binding does not
currently ship the richer Python-style app-facing layer:

- task decorators
- caller-side `TaskResult`
- task registry
- worker CLI / managed task runner ergonomics

That is a product choice, not automatically a defect.

## What we do next

The next parity push should happen in this order:

1. move maintained bindings toward one source-of-truth repo shape
2. keep the core-runtime checklist green across all maintained bindings
3. decide which bindings should grow the richer task-framework layer
4. only then start building that richer layer beyond Python

That order keeps us from rebuilding the same feature nine times while
the repo shape is still fighting us.

## Repo shape

The current many-repos-plus-submodules setup made parity work much
harder than it needed to be.

The likely future shape is:

- main repo stays the source of truth
- maintained bindings live together in `packages/`
- one feature can ship as one coordinated change
- separate public binding repos, if kept, become mirrors or release
  artifacts rather than the primary place humans edit

That is the direction tracked in the roadmap.
